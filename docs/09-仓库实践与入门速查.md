## 2. 为什么使用 TypeScript

JavaScript 是**动态类型**语言：灵活易学，但大型项目中类型错误往往到**运行时**才暴露。

TypeScript 在 JS 之上增加**静态类型检查**，常见收益：

- **兼容 JavaScript**：现有 JS 可逐步迁移。
- **更高可预测性**：在编译期发现误用 API、拼写、`null`/`undefined` 等问题。
- **更好的工具支持**：补全、重构、跨文件跳转更可靠。
- **适合中大型项目**：模块、接口、泛型等便于组织代码。

类型信息**不会进入运行时**；`tsc`（或 `esbuild`、`swc` 等）负责检查与转译。详见 [00-定位与目标](./00-定位与目标.md)。

## 3. 快速开始

### 3.1 安装 TypeScript

```bash
# 全局（可选，便于直接调用 tsc）
npm install -g typescript

# 项目内（推荐）
npm install -D typescript
# 或
pnpm add -D typescript
```

### 3.2 初始化配置

```bash
npx tsc --init
```

生成 `tsconfig.json`。常用选项说明见 [07-工程与配置](./07-工程与配置.md)，勿照搬过时模板（如固定 `target: es5` + `module: commonjs`）；应按运行环境选择 `target`、`module`、`moduleResolution`。

### 3.3 编译与检查

```bash
npx tsc              # 按 tsconfig 编译
npx tsc --noEmit     # 仅类型检查，不输出文件
npx tsc --watch      # 监听模式
```

常用 CLI 标记示例：

| 标记 | 作用 |
|------|------|
| `--strictNullChecks` | 单独开启严格空值检查（通常直接开 `strict`） |
| `--noEmit` | 只检查、不生成 JS |

### 3.4 运行本仓库 `base/` 示例

```bash
cd base
pnpm install     # 推荐；亦可用 npm install
pnpm run dev     # nodemon + ts-node 运行 src/interface-class.ts
pnpm start       # tsc -w 监听编译
```

## 4. `tsconfig.json` 速查

### 4.1 顶层字段

| 字段 | 说明 |
|------|------|
| `include` / `exclude` | glob：`**` 任意目录，`*` 任意文件名 |
| `files` | 显式文件列表 |
| `compilerOptions` | 编译与类型检查选项 |

### 4.2 常用 `compilerOptions`

| 选项 | 说明 |
|------|------|
| `target` | 输出 JS 语言版本 |
| `module` | 模块格式（与 Node / 打包器一致） |
| `lib` | 注入的环境类型库 |
| `outDir` / `outFile` | 输出目录或单文件（`outFile` 受 `module` 限制） |
| `allowJs` / `checkJs` | 纳入并检查 JS 文件 |
| `removeComments` | 去掉注释 |
| `noEmit` | 不生成输出 |
| `noEmitOnError` | 有错时不生成文件 |
| `strict` | 严格检查总开关 |
| `skipLibCheck` | 跳过 `.d.ts` 库检查以加速 |

严格子选项（`strict` 开启时通常一并生效）：`noImplicitAny`、`strictNullChecks`、`strictFunctionTypes`、`noImplicitThis` 等。完整列表见 [TSConfig Reference](https://www.typescriptlang.org/tsconfig/)。

### 4.3 推荐起步配置（示意）

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true
  },
  "include": ["src/**/*"]
}
```

## 5. 类型速查（与体系文档对照）

以下条目在 [01-基础](./01-基础.md)、[02-日常类型](./02-日常类型.md) 中已有展开，此处仅作索引。

| 类别 | 类型 / 概念 | 要点 |
|------|-------------|------|
| 原始 | `boolean`、`number`、`string`、`bigint`、`symbol` | 日常使用小写原始类型 |
| 集合 | `T[]`、`Array<T>`、元组 `[T, U]` | 元组为固定长度、各位置类型可不同 |
| 特殊 | `any`、`unknown`、`never`、`void` | `unknown` 为类型安全的顶层；`never` 为底类型 |
| 空值 | `null`、`undefined` | `strictNullChecks` 下需显式处理 |
| 对象 | `object`、接口、可选 `?`、索引签名 | 见 [02-日常类型](./02-日常类型.md) |
| 组合 | 联合 `\|`、交叉 `&`、字面量 | 联合为「其一」；交叉为「同时满足」 |
| 别名 | `type Name = ...` | 类型运算、联合多用 `type` |
| 断言 | `value as Type` 或 `<Type>value` | 告知编译器更具体的类型，勿滥用 |
| 只读 | `readonly`、`Readonly<T>`、`ReadonlyArray<T>` | 只作用于属性/数组元素；与 `const` 不同，见下 |

**`readonly` 与 `const`**

- `readonly`：属性不可重新赋值，对象其它属性仍可改。
- `const`：变量绑定不可改，对象属性仍可改。

**类型断言两种写法**

```ts
const len = (str as string).length;
const len2 = (<string>str).length; // JSX 文件中避免 <>
```

## 7. 声明空间（类型 vs 值）

TypeScript 中存在两个主要「声明空间」：

| 空间 | 常见成员 | 说明 |
|------|----------|------|
| **类型** | `interface`、`type`、`class`（类型侧） | 仅编译期存在 |
| **值** | `const`、`let`、`var`、`class`（值侧）、函数 | 会进入 JS 输出 |

`class` 在类型空间与值空间**同时**存在，因此既可作类型又可 `new`。

## 8. 模块与命名空间

### 8.1 模块（推荐）

- **全局脚本**：无 `import`/`export` 的文件，符号落在全局（应避免在新代码中使用）。
- **ES 模块**：含 `import`/`export` 的文件拥有模块作用域；`compilerOptions.module` 决定输出格式。

模块解析路径：

1. 相对路径 `./`、`../`
2. 非相对：Node 按 `package.json` 的 `exports`/`main`/`types` 等解析

详见 [06-类与模块](./06-类与模块.md)。

### 8.2 命名空间（历史用法）

`namespace` 曾用于在单文件内分组；现代项目优先 **ES 模块**。若维护旧代码，见官方 [Namespaces](https://www.typescriptlang.org/docs/handbook/namespaces.html)。

### 8.3 动态导入

```ts
const mod = await import('./feature');
```

Webpack 等打包器中的 `require.ensure` 属于工具链 API，非 TS 语言本身。

## 9. 与打包器集成（Webpack 简述）

Handbook 不展开各打包器细节；典型组合如下：

```bash
pnpm add -D webpack webpack-cli typescript ts-loader
```

在 `webpack.config` 中对 `.ts` 使用 `ts-loader`（或 `babel-loader` + `@babel/preset-typescript`），并令 `tsconfig` 的 `noEmit: true` 由 Webpack 负责输出。现代项目也常选 **Vite**、**esbuild**、**swc** 等，原则相同：**类型检查**与**转译**可分离（例如 `tsc --noEmit` + `esbuild` 打包）。

## 10. 在 Node 中使用 TypeScript

```bash
npm init -y
npm install -D typescript @types/node ts-node nodemon
npx tsc --init
```

开发时监听示例：

```bash
nodemon --watch 'src/**/*.ts' --exec 'npx ts-node' src/index.ts
```

生产环境常见做法：`tsc` 编译到 `dist/` 后 `node dist/index.js`，或使用 `tsx` / 预编译部署。

## 11. 依赖管理（npm）

| 类型 | 说明 |
|------|------|
| `dependencies` | 运行时需要的包 |
| `devDependencies` | 仅开发、构建、测试需要 |
| `peerDependencies` | 由宿主环境提供的包；安装时**不会**自动安装，用于插件/框架声明兼容版本 |

类型定义包通常放在 `devDependencies`：`@types/node`、`@types/react` 等。

## 12. JavaScript 项目迁移到 TypeScript

建议步骤（与官方 [Migrating from JavaScript](https://www.typescriptlang.org/docs/handbook/migrating-from-javascript.html) 一致）：

1. 添加 `tsconfig.json`（可先 `allowJs: true`）。
2. 将 `.js` 逐步改为 `.ts`/`.tsx`；暂时可用 `any` 或类型断言减少报错。
3. 为关键 API 补充类型注解，修复检查错误，**逐步减少 `any`**。
4. 为无类型的第三方库添加声明（`@types/*` 或自建 `.d.ts`）。

## 13. 联合类型与交叉类型

| 种类 | 含义 | 可访问成员 |
|------|------|------------|
| **联合** `A \| B` | 值为 A 或 B 之一 | 仅能安全使用**共有**成员（需先收窄，见 [03-收窄](./03-收窄.md)） |
| **交叉** `A & B` | 同时满足 A 与 B | 可访问两者成员（冲突属性需兼容） |

**`interface` vs `type` 选型**：需要声明合并、`implements` 的公共对象形状 → 优先 `interface`；联合、交叉、条件/映射类型 → 多用 `type`。
