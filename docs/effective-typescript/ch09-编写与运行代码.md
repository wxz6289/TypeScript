# 第 9 章 · 编写与运行代码（Writing and Running Your Code）

对应原书第 9 章（公开资料提及 **Item 72: Prefer ECMAScript Features to TypeScript Features**、运行时重建类型、环境与性能等）。

---

## 1. 核心原则：运行时归 TC39，类型层归 TS

随着 ES 标准演进，**类、模块、异步**等应由标准 JavaScript 语义承载；TS 专注在**静态类型**上创新（模板字面量类型、推断等）。  
遗留的 TS-only 运行时特性（若仍存在）应谨慎评估——长期看会增加互操作与教学成本。

---

## 2. 何时「在运行时重建类型」

- 序列化/反序列化、JSON Schema、配置加载：需要 **schema 校验**（如 zod/io-ts）而不只依赖断言。  
- 类型无法保证跨边界（网络、磁盘）为真。

```ts
// 类型不能替运行时校验「线上返回的 JSON」
declare const raw: unknown;
// 应用校验器后再进入业务类型
```

---

## 3. 为运行环境建模：`lib` 与 DOM 类型

- Node / Browser / Worker 选择匹配的 `lib` 与 `types`，避免「在 Node 里误用 DOM API」类错误。  
- 对 `globalThis` 扩展使用 `declare global`。

---

## 4. 类型检查 vs 单元测试

- 类型覆盖**组合爆炸**的不变量有优势；测试覆盖**具体例子与回归**。  
- 二者互补：不要把所有断言都写成 `as`；也不要假设测试能替代类型。

---

## 5. 编译性能

- 控制 `include` 范围、启用 `skipLibCheck`（权衡）、拆分工程引用（Project References）。  
- 大仓库中「全量 `tsc`」与「IDE 语言服务」瓶颈不同，需分别 profiling。

---

## 6. 自检清单

- [ ] 新代码是否优先使用标准 ES 语法而非历史 TS 独有运行时特性？  
- [ ] 对外部输入是否始终有运行时校验层？
