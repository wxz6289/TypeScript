# 第 1 章 · 认识 TypeScript（Getting to Know TypeScript）

## 1. 本章在解决什么问题

让读者建立**正确预期**：TypeScript 与 JavaScript 的分工、编译器选项的影响、类型与运行时的关系、结构类型心智模型，以及为何要尽量远离 `any`。

---

## 2. 心智模型速记

| 要点 | 说明 |
|------|------|
| TS 是 JS 的超集 | 语法上兼容；**类型信息在擦除后不进入运行时**（除非你用显式反射式元数据等额外机制）。 |
| 类型检查 ≠ 代码生成 | `tsc` 可只做检查（`noEmit`），也可向下编译；**改类型不应指望改变运行逻辑**。 |
| 结构类型 | 「能否赋值」看**形状**是否兼容，而非声明名是否相同。 |
| 了解你的编译选项 | `strict` 家族决定大量日常体验；团队应对齐 `tsconfig`。 |
| 限制 `any` | `any` 会关闭检查；应视为**技术债**，范围越小、存在时间越短越好。 |

---

## 3. 结构类型小例

```ts
interface Vector2D {
  x: number;
  y: number;
}

function length(v: Vector2D) {
  return Math.hypot(v.x, v.y);
}

const p = { x: 3, y: 4 };
length(p); // 变量 p 的结构满足即可

// 字面量「新鲜」对象可能触发多余属性检查，与「结构类型」并存——详见 TS 手册 excess property checks
```

**实践**：设计 API 时，用「最小必要形状」作为参数类型，比绑死具体类名更灵活。

## 4. `any` 的替代思路（预告后文）

- 边界未知：`unknown` + 收窄。  
- 第三方过宽：局部封装 + 更窄的包装类型。  
- 渐进迁移：`// @ts-expect-error` 窄范围使用（第 5 章会展开）。
