# 第 4 章 · 类型设计（Type Design）

对应原书第 4 章（含 **Item 29: Prefer Types That Always Represent Valid States** 等，见原书）。

---

## 1. 本章一句话

**让不可能状态在类型里不可表示**——比写注释、加运行时校验更便宜、更可靠。

---

## 2. 反模式：布尔旗标堆叠

```ts
// 容易非法：loading=true 且 error 有值、或 data 与 error 同时存在等
interface Bad {
  loading: boolean;
  error?: string;
  data?: User;
}
```

---

## 3. 正模式：状态机式联合

```ts
interface User {
  id: string;
  name: string;
}

type RemoteUser =
  | { state: 'idle' }
  | { state: 'loading' }
  | { state: 'error'; message: string }
  | { state: 'success'; data: User };

function render(u: RemoteUser) {
  switch (u.state) {
    case 'success':
      return u.data.name;
    case 'error':
      return u.message;
    default:
      return '…';
  }
}
```

---

## 4. API 形状：参数对象 vs 长参数列表

- 可选参数增多时，用 **单一 options 对象** + 明确类型，便于演进。  
- 为常用路径提供 **defaults**（解构默认值），保持调用点整洁。

---

## 5. 避免在类型与文档中重复同一事实

- 若类型已表达「非空字符串」，文档不必再抄一遍；文档应写**业务约束**（为何必须非空）。  
- 使用 **TSDoc** 描述行为与不变量（第 8 章会再呼应）。
